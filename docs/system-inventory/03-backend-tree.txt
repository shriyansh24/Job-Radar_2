## App Module Grouping

| Group | Paths | Purpose |
|---|---|---|
| Runtime core | `D:/jobradar-v2/backend/app/main.py`, `config.py`, `database.py`, `dependencies.py`, `shared` | Boot, middleware, auth/session resolution, DB access, logging |
| Business domains | `D:/jobradar-v2/backend/app/auth` through `D:/jobradar-v2/backend/app/vault` | API routes, service logic, models, schemas |
| Background execution | `D:/jobradar-v2/backend/app/workers` | Periodic jobs, orchestration, maintenance |
| Migration history | `D:/jobradar-v2/backend/app/migrations` | Schema change source of truth |
| Shared AI/NLP support | `D:/jobradar-v2/backend/app/nlp`, plus LLM-related files in resume/interview/copilot/enrichment | Model selection, prompting, generation helpers |
| Test surface | `D:/jobradar-v2/backend/tests` | Unit/integration/contract/security verification |

## Source-Of-Truth Notes

- The backend is a modular monolith, not a service mesh or microservice split.
- Domain services are the effective source of truth for business logic because there is no repository abstraction layer.
- Alembic migrations and SQLAlchemy models jointly define persistence. Neither alone is fully sufficient for refactor safety.
