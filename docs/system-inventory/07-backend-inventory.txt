# Backend Inventory

Root: `D:/jobradar-v2/backend`

## Backend Architecture Summary

- Framework: FastAPI
- Entry point: `D:/jobradar-v2/backend/app/main.py`
- Data access: async SQLAlchemy direct from services and workers
- Migration system: Alembic in `D:/jobradar-v2/backend/app/migrations`
- Auth model: JWT access/refresh tokens with cookie support and bearer-token compatibility
- Worker model: APScheduler started in-process from API lifespan
- Repository layer: none

## Data Access Pattern

| Area | Status |
|---|---|
| Repository layer | Not present |
| Unit of work | Not centralized |
| Transaction boundary | Usually inside each service or worker function via `db.commit()` and `db.refresh()` |
| Query abstraction | Domain service methods and worker functions |
| Test DB strategy | SQLite metadata bootstrap in `tests/conftest.py`, not full Postgres migration replay by default |

## Backend Risk Notes

1. `main.py` and `workers/scheduler.py` make the API server the scheduler host.
2. `scraping` and `auto_apply` are the highest-risk domains because they combine long side-effect chains, browser tooling, and external anti-bot behavior.
3. Auth changes ripple across `dependencies.py`, `auth/service.py`, and the frontend refresh interceptor in `frontend/src/api/client.ts`.
4. Postgres-specific behavior is not fully represented in SQLite-backed tests.
5. `vault/router.py` exposes update handlers that `vault/service.py` does not currently implement.
