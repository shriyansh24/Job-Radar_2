## Important Record Lifecycles

### User and profile lifecycle

1. `users` is created via auth registration.
2. `user_profiles` is created lazily or during onboarding/profile update flows.
3. `saved_searches`, `user_integration_secrets`, `resume_versions`, `auto_apply_profiles`, `notifications`, and other user-owned records accumulate around the same `user_id`.
4. Account deletion uses admin-style data clearing and then deletes the user.

### Job ingestion lifecycle

1. A scrape run is started and logged in `scraper_runs`.
2. Targeted or source-specific fetch attempts are tracked in `scrape_attempts`, optionally against `scrape_targets`.
3. Parsed job records are normalized into `jobs`.
4. Jobs may later be grouped into `canonical_jobs` and bridged through `raw_job_sources`.
5. Source quality and health are summarized in `source_registry` / `source_check_log`.

### Application execution lifecycle

1. A job becomes an `application` manually or from job detail/create flows.
2. Each status change creates `application_status_history`.
3. Optional follow-up reminder rows are attached to the application.
4. Outcome capture creates or updates one `application_outcomes` row.
5. Company-level statistics roll into `company_insights`.

### Preparation artifact lifecycle

1. Resume upload creates `resume_versions`.
2. Tailoring and council flows read resume versions plus job context; tailored text is returned, while persisted cover-letter-like artifacts live in `cover_letters`.
3. Interview sessions persist generated questions, answers, and scores in `interview_sessions`.
4. Salary research caches results in `salary_cache`.

### Automation lifecycle

1. `auto_apply_profiles` and `auto_apply_rules` define eligibility and submission behavior.
2. Execution writes `auto_apply_runs`.
3. ATS interactions can learn `field_mapping_rules`.
4. Dedup state is tracked in `application_dedup`.

## Soft / Inferred Relationships

| Relationship | Basis | Why It Matters |
|---|---|---|
| `jobs.company_name` / `jobs.company_domain` -> `companies` | String/domain matching, not FK | Company normalization drift can break networking/outcomes/analytics assumptions |
| `company_insights.company_name` -> `companies` | String match | No enforced referential integrity |
| Networking company searches -> `companies` / `jobs` | Inferred business logic, not schema | Referral quality depends on normalization consistency |
| Search expansion tables -> frontend admin surface | Intended feature link, not complete implementation | UI/API currently drift from stored model state |

## Migration Interpretation Notes

- The migration history shows a project that evolved through feature branches and later consolidation.
- The most trustworthy way to reason about the current schema is:
  1. inspect the model file
  2. inspect the creating migration
  3. inspect later altering migrations
  4. inspect the service that reads/writes the table
- For fresh database bootstrap confidence, a clean Postgres replay test is still needed.
