# Runtime Topology

Root scope: whole system runtime across browser, frontend container/dev server, backend API, DB, Redis, workers, and CI.

## Topology Summary

- Browser clients talk to the React SPA.
- The SPA calls the backend under `/api/v1`.
- The backend is a single FastAPI process that also starts APScheduler jobs.
- PostgreSQL is the primary persistent store.
- Redis is provisioned as a runtime dependency but is not the backbone of request-time coordination.
- Scraping and auto-apply can fan out into external websites, browser automation, and ATS-specific adapters.

## Runtime Components

| Component | Runtime Location | Primary Files | Responsibilities | Notes |
|---|---|---|---|---|
| Browser UI | User browser | `frontend/src/main.tsx`, `frontend/src/App.tsx` | Render routes, manage auth/session state, drive API calls | Query and UI state are client-side |
| Frontend dev server | Frontend container or local node process | `frontend/vite.config.ts`, `frontend/start-dev.cjs` | Dev-time bundling, proxying, HMR | Proxy target is misconfigured for in-container dev |
| Frontend runtime server | Frontend container (base compose) | `frontend/Dockerfile` Nginx stage | Serves built SPA and proxies `/api` to backend | Production-ish only inside same Docker network |
| Backend API | Backend container or local `uvicorn` process | `backend/app/main.py`, `backend/Dockerfile` | HTTP API, auth, business logic, scheduler host | Modular monolith |
| DB | Postgres container | `docker-compose.yml` | Persistent relational storage | PostgreSQL-first path |
| Redis | Redis container | `docker-compose.yml`, `infra/redis/tls/.gitkeep` | Cache/broker-like runtime service | Not heavily used for distributed coordination yet |
| Scheduler | Same backend process | `backend/app/workers/scheduler.py` | Registers recurring jobs | Not externally separated |
| Worker functions | Same backend process | `backend/app/workers/*.py` | Batch scraping, enrichment, alerts, follow-up, auto-apply | Invoked by APScheduler |
| CI | GitHub Actions | `.github/workflows/*.yml` | Lint, test, build, static analysis | No deployment job |

## Local Development Flow

### Containerized

1. `docker-compose.yml` starts backend, frontend, postgres, redis.
2. `docker-compose.dev.yml` overlays bind mounts and reload commands.
3. Backend starts, applies migrations, then runs Uvicorn.
4. Frontend either serves a static build through Nginx or a Vite dev server.
5. Browser talks to frontend on `3000` in base compose or `5173` in dev overlay.

### Manual

1. Postgres is started manually per README/CLAUDE guidance.
2. Backend is run with `uv`.
3. Frontend is run with `npm run dev`.

## Ports, Volumes, And Networks

| Service | Ports | Volumes | Network Assumption |
|---|---|---|---|
| Frontend base compose | `3000:80` | none in base file | Same compose network as backend |
| Frontend dev overlay | `5173:5173` | bind-mounted source | Same compose network, but proxy currently targets localhost |
| Backend | `8000:8000` | bind mount in dev overlay | Same compose network as DB/Redis |
| Postgres | `5432:5432` | named volume `pgdata` | Same compose network as backend |
| Redis | `6379:6379` | runtime data/TLS placeholder mounts | Same compose network as backend |

## Production Assumptions Inferable From The Repo

- The likely intended production shape is still:
  - one frontend runtime
  - one backend runtime
  - one Postgres instance
  - one Redis instance
- The repo does not show:
  - horizontal scheduler coordination
  - dedicated worker process separation
  - infrastructure-as-code for a production target
  - deployment rollouts or release automation

## Topology Risks

1. The backend API and scheduler share one process, so scaling API replicas would duplicate jobs.
2. Frontend dev proxy settings do not match the containerized dev topology.
3. Runtime docs disagree on DB version, port, and credentials.
4. SSE and in-memory events are single-instance scoped.
5. Browser-heavy scraping/auto-apply paths are more demanding than the current generic runtime definition suggests.
