## 1. Phase 7A Goal

Phase 7A adds the first 5 foundational modules needed to move JobRadar from a job discovery dashboard toward a durable job intelligence system:

1. Company Intelligence Registry
2. Search Expansion Engine
3. Validated Source Cache
4. Canonical Jobs Pipeline
5. Application Tracker

The goal is to extend the current system safely, incrementally, and compatibly.

## 4. Locked Architecture Decisions

### M1 — Company Intelligence Registry
- Introduce canonical company identity.
- Use deterministic company records.
- Prefer URL-pattern and public-careers-page detection first.
- Manual overrides must always survive automated refresh.
- Do not assume expensive external company datasets.
- Company identity must be reusable by source validation and canonical jobs.

### M2 — Search Expansion Engine
- V1 is deterministic and rule-based.
- Use stored query AST / structured expansion representation.
- No embedding-driven expansion in V1 unless explicitly approved later.
- No LLM-generated query expansion in V1.
- Query generation must be traceable, previewable, and source-aware.
- Prevent query explosion and redundant expansions.

### M3 — Validated Source Cache
- Source health must be persisted.
- Source health writes must be batched/coalesced for SQLite safety.
- Track freshness, failure patterns, and revalidation state.
- Handle transient and permanent failures differently.
- Manual override and force-revalidate must exist.
- Scheduler must eventually prefer healthier sources.

### M4 — Canonical Jobs Pipeline
- Introduce canonical layer without replacing current `jobs` immediately.
- Coexistence-first design is mandatory.
- No flag-day migration.
- Canonical identity must be deterministic and stable.
- Provenance must be preserved.
- Raw source records and canonical records must be separable.
- Existing `jobs` flows must keep working during transition.

### M5 — Application Tracker
- User workflow state must not live in `jobs`.
- Tracker state must live in a separate applications/tracker model.
- User-authored state always wins over scraper refreshes.
- Must support graceful fallback to legacy `job_id` before M4 is complete.
- Later prefer `canonical_job_id`.
- Must be future-ready for browser autofill and auto-apply events.

## 5. Ownership Rules

### System-owned data
These may be created or updated by scrapers/pipelines:
- source-derived job fields
- source health fields
- ATS/provider detection fields
- canonical merge/provenance fields
- freshness/validation timestamps
- quality scores generated by the system

### User-owned data
These must never be overwritten by scraper refreshes:
- saved/applied state
- application status
- notes
- reminders
- follow-up dates
- tags
- user annotations
- custom workflow fields

If there is a conflict, user-owned state wins.

## 6. Migration Rules

- Additive migrations only.
- Do not drop or rename existing production-facing fields in Phase 7A.
- New tables may be introduced.
- Existing reads may coexist with new reads.
- Dual-read / coexistence is preferred over hard cutover.
- Legacy compatibility must be maintained until explicitly retired in a future phase.
- Any migration must include rollback thinking and compatibility validation.
